# 호스트 런타임(자동화) 엔진 — 옵트인 설정

rhwp-mcp-server는 기본적으로 번들된 **WASM 엔진**으로 동작합니다(설치 불필요,
모든 플랫폼). 그 위에, 로컬에 설치된 한글 워드프로세서의 **자동화 표면**을
구동하는 **호스트 런타임 엔진**을 *옵트인*으로 제공합니다. 이 엔진은 실제
문서 객체 모델을 다루므로 셀 기하 등에서 더 높은 충실도를 목표로 하지만,
**대화형 윈도우 데스크톱에서만** 동작합니다(ADR-0008).

> 이 엔진은 기본 OFF입니다. 켜지 않으면 동작이 전혀 바뀌지 않으며, 기존
> WASM 경로가 그대로 쓰입니다.

## 동작 조건 (셋 다 충족해야 활성)

1. **옵트인** — 환경변수 `RHWP_COM=1`. 이 엔진은 데스크톱에서 살아있는
   문서를 구동하므로, 절대 묵시적 기본값이 될 수 없습니다. 명시적으로 켜야
   합니다.
2. **설치/등록** — 윈도우에 한글 워드프로세서가 설치되어 자동화 객체가
   레지스트리에 등록되어 있어야 합니다(`hwp_engine_status`가 `AVAILABLE`로
   보고).
3. **핸드셰이크 성공** — Python 헬퍼의 `ping`이 성공해야 합니다(파이썬 +
   자동화 래퍼가 임포트되고 등록 확인). 등록은 됐지만 헬퍼에 도달하지 못하면
   여전히 비활성으로 보고되고, 기존 자동 WASM 폴백이 적용됩니다(ADR-0007).

## 사전 준비

```powershell
# 1) 한글 워드프로세서가 설치된 대화형 윈도우 데스크톱
# 2) Python 런타임 (PATH 에 있거나 RHWP_PYTHON 으로 경로 지정)
# 3) 자동화 래퍼 (선택 의존 — npm 패키지가 자동 설치하지 않음)
pip install pyhwpx
```

`python` 외 다른 실행기를 쓰면 `RHWP_PYTHON`으로 지정합니다(예: `py`,
절대 경로 등). 헬퍼 스크립트는 패키지에 동봉됩니다(`python/rhwp_hwp_helper.py`).

## 켜기

MCP 클라이언트가 서버를 spawn할 때 환경변수를 전달합니다.

```jsonc
{
  "mcpServers": {
    "rhwp": {
      "command": "npx",
      "args": ["-y", "rhwp-mcp-server"],
      "env": { "RHWP_COM": "1" }
    }
  }
}
```

상태 확인:

```text
hwp_engine_status 호출 → engines[].detail 에 opt-in 과 python-wrapper 가용성이
표시됩니다. active 가 'com' 이면 활성, 'wasm' + fallback_reason 이면 폴백 중.
```

## 보안 모듈 (별도 옵트인, 기본 OFF)

자동화 표면은 파일 접근마다 확인 프롬프트를 띄울 수 있습니다. 허용형 파일경로
검사 모듈을 등록하면 그 프롬프트가 억제되어 무인 읽기/쓰기가 가능하지만, 이는
**안전 프롬프트를 의도적으로 완화**하는 행위입니다. 그래서 기본 OFF이며,
`RHWP_COM_REGISTER_MODULE=1`을 추가로 설정한 경우에만 등록됩니다. 본 프로젝트는
벤더의 샘플 검사 바이너리를 재배포하지 않습니다 — 등록은 호스트에 이미 설치된
것에 대해 래퍼/설정 단계에서 처리됩니다.

## 검증 (대화형 데스크톱에서 직접)

서버를 띄우기 전에 실제 구동을 점검하려면:

```powershell
$env:RHWP_COM = "1"
npm run spike:com   # ping → create_blank → quit
```

> ⚠ `spike:com` 은 **살아있는 자동화 객체를 생성**합니다. 비대화형 컨텍스트
> (CI, 헤드리스 셸, 에이전트 세션)에서는 객체 생성이 **무한 대기로 블록**됩니다.
> 반드시 본인이 대화형 데스크톱에서 직접 실행하세요.

## 폴백 의미 (변하지 않음)

옵트인이 꺼져 있거나, 호스트가 미설치/미등록이거나, 핸드셰이크가 실패하면
서버는 조용히 WASM 엔진으로 폴백합니다(ADR-0007). 즉, 이 엔진을 켜도 실패는
호출자에게 typed 에러나 자동 폴백으로만 나타나며, 절대 멈추지 않습니다 —
블록 가능성이 있는 명령은 Node가 명령별 타임아웃을 걸어 typed `COM_TIMEOUT`로
바꾸고 헬퍼를 종료합니다.
